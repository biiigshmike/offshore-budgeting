# Marina Natural Language Architecture Diagnostic

Date: 2026-04-29
Scope: Inspection-only. No business math or route behavior changed.

## 1. End-to-End Architecture Map

### User text entry points
- Main assistant input: `HomeAssistantFoundation.interpretPrompt(_:)` in `OffshoreBudgeting/CoreViews/Home/HomeAssistantFoundation.swift`.
- Clarification response input (deterministic NLQ mode): `handleNLQClarificationInput(_:payload:)`.
- Suggestion tap replay path: `handleSuggestionTap(_:)` -> `runQuery(...)` or NLQ clarification branch.

### Primary orchestration switch
- `interpretPrompt(_:)` checks `marinaNLQv1Enabled`.
- If `true`: deterministic Marina NLQ path (`MarinaNLQPipeline`) is authoritative.
- If `false`: router/model path (`MarinaLanguageRouter`) runs, with heuristic parser fallback closure.

### Interpretation layers
- Foundation model path:
  - `MarinaLanguageRouter.interpret(...)`
  - `MarinaFoundationModelsService.interpret(...)` (schema-driven structured output)
  - `MarinaStructuredIntentPlanBuilder.buildRequest(...)`
- Deterministic Marina NLQ path:
  - `MarinaNLQPipeline.run(...)`
  - `MarinaNLQNormalizer` -> `MarinaIntentInterpreter` / `MarinaMetricMapper`
  - `MarinaNLQCandidateExtractor` + `MarinaNLQResolver`
- Classic assistant fallback path:
  - `heuristicInterpretedRequest(for:)`
  - `HomeAssistantCommandParser` / `resolvedPlan(for:)`
  - `HomeAssistantTextParser`, `HomeAssistantIntentBuilder`, `HomeAssistantResolutionPipeline`

### Execution/math layers
- Main deterministic financial execution engine: `HomeQueryEngine.execute(...)`.
- Marina NLQ execution adapter: `MarinaNLQAggregationEngine.aggregate(...)` (maps NLQ metric -> `HomeQuery` and calls `HomeQueryEngine`).
- Additional path-specific builders (presentation and certain diagnostics) in `HomeAssistantFoundation`, but math source-of-truth remains `HomeQueryEngine`.

### Response + clarification layers
- Router/model mode:
  - `handleInterpretedRequest(...)`
  - `handleResolvedPlan(...)`, `handleMarinaClarification(...)`
  - clarification chips via `presentClarification(...)` and `clarificationSuggestions(...)`
- NLQ mode:
  - `MarinaNLQPipelineResult` (`answer` / `clarification` / `recovery`)
  - `handleNLQPipelineResult(...)`
  - final answer text through `MarinaNLQResponseBuilder` then persona formatting.

## 2. Routing Decision Table

| Condition | Selected route | Notes |
|---|---|---|
| `marinaNLQv1Enabled == true` | Deterministic NLQ (`MarinaNLQPipeline`) | Router/model query and model clarification are explicitly blocked in `handleInterpretedRequest` for NLQ-authoritative mode. |
| `marinaNLQv1Enabled == false` + model unavailable | Heuristic fallback | `MarinaLanguageRouter`: immediate `heuristicFallback()`.
| Model returns valid query | Model query, unless overridden | `resolveModelQuery(...)` may prefer heuristic query via `shouldPreferHeuristicQuery(...)`.
| Model returns clarification | Clarification, unless overridden | `shouldPreferHeuristicFallback(...)` can choose executable fallback query.
| Model unresolved or throws error | Fallback or clarification | Query-like prompts go through `resolveQueryLikeFailure(...)` to fallback query/clarification.
| Query-like model failure + fallback broad generic plan | Clarification can be preserved | `shouldPreserveClarificationOverFallback(...)` prevents silent broad fallback for grouped/ranked/unsupported-average prompts.

Key finding: fallback can replace model output in multiple branches (`model clarification`, `model query`, `model error/unresolved`) based on specificity and follow-up heuristics.

## 4. Model->Executable Adapter Fidelity

Observed fidelity risk points:
- `targetTypeRaw` exists in structured output but execution mostly depends on normalized metric + `targetName` + later canonicalization.
- `periodUnitRaw` is preserved on `HomeQueryPlan`, but NLQ aggregation path currently sets `periodUnit: nil` on mapped queries.
- Presentation style is not a first-class structured query field in `MarinaStructuredQueryIntent`.
- Comparison range parsing can be inferred from prompt even when model omits explicit comparison dates.
- Heuristic fallback can replace valid model output if fallback is deemed more specific.

Fields requested for inspection status:
- target: represented (`targetName`, optional `targetTypeRaw`) -> can be canonicalized later.
- entity type: represented but weakly enforced across all branches.
- date range: represented + parser-derived fallback.
- comparison date range: represented + parser-derived fallback.
- operation: represented by metric mapping.
- presentation style: not explicitly represented in model intent schema.

## 6. Silent Override Risk Findings

1. Valid model query can be replaced by fallback query.
- In `MarinaLanguageRouter.shouldPreferHeuristicQuery(...)`, fallback wins for higher specificity, missing model date range, follow-up metric continuity, or follow-up target carryover.

2. Model clarification can be replaced by fallback executable query.
- In `shouldPreferHeuristicFallback(...)`, non-actionable or low-confidence-only clarifications can be bypassed.

3. Model unresolved/error in query-like prompts may still execute fallback.
- `resolveQueryLikeFailure(...)` can execute fallback query directly unless clarification-preservation guard applies.

4. Preservation guard exists but is selective.
- `shouldPreserveClarificationOverFallback(...)` blocks broad fallback for grouped/ranked/unsupported-average prompt patterns.

## 7. Proposed Debug Trace Type (Design + Additive Type)

A minimal additive trace schema is included in code as `MarinaExecutionTrace` with:
- prompt/original + normalized
- model availability
- model output summary
- model validation result
- fallback usage + reason
- normalized metric/operation
- target raw/resolved
- date/comparison ranges
- aggregation path/result summary
- response path + final outcome

Insertion points (no behavior redesign):
1. `MarinaLanguageRouter.interpret(...)`:
- before availability check
- after model response
- after validation
- on fallback preference decisions
- on final route selection
2. `MarinaNLQPipeline.run(...)` / `resolveClarificationResponse(...)`:
- after normalization
- after resolution outcome
- after aggregation
3. `HomeAssistantFoundation.handleInterpretedRequest(...)` and `handleMarinaClarification(...)`:
- final presentation branch (answer/clarification/recovery/unresolved)
